# Учёт оплат, проектов и закрывающих актов

Мини-система для digital-агентства: из банковской выписки строит понятную картину
**проект / юрлицо → оплаты → этапы работ → статус закрывающих актов**. Видно, кто
заплатил, по какому проекту и направлению, отправлен ли акт, подписан ли, где долги
по закрытию и что требует внимания.

> Тестовое задание. Исходное ТЗ — [`docs/brief/TASK.md`](docs/brief/TASK.md),
> мок-данные — банковская выписка в [`docs/brief/`](docs/brief/).

Ключевая идея: **не каждая операция выписки — это оплата клиента за проект.** Из 47
операций выписки только 24 — выручка по проектам (1 405 820,00 ₽); остальное (налоги,
зарплаты, аренда, банковские комиссии, депозиты, оплаты субподрядчикам) система
отделяет как шум. Эта классификация сходится с итогами выписки до копейки.

## Быстрый запуск (Docker)

```bash
docker compose up --build      # → http://127.0.0.1:8099
```

Поднимает PostgreSQL + Laravel API + nginx (отдаёт собранный Vue SPA и проксирует
`/api`). Миграции и сиды из приложенной выписки выполняются автоматически. Открыть
дашборд: **http://127.0.0.1:8099**.

> `docker-compose.yml` настроен под текущий деплой: `APP_ENV=production`, сервис `web`
> слушает `127.0.0.1:8099` (наружу отдаётся через Cloudflare Tunnel). Рабочая версия —
> **https://business-up.dm312sv.online**.

## Локальная разработка (без Docker)

```bash
# Backend (Laravel API на SQLite)
cd backend
cp .env.example .env            # для локали поменяйте DB_CONNECTION на sqlite
composer install
php artisan migrate --seed      # парсит выписку и наполняет БД
php artisan serve               # http://localhost:8000

# Frontend (Vue SPA), в отдельном терминале
cd frontend
npm install
npm run dev                     # http://localhost:5173, /api проксируется на :8000
```

Прогон тестов: `cd backend && php artisan test` (35 тестов на SQLite).
Импорт другой выписки: `php artisan statement:import path/to/statement.pdf`.

## Архитектура

Данные → бизнес-логика → отображение разнесены по явным слоям:

```
PDF-выписка
  └─ StatementParser + OperationClassifier  →  bank_operations (сырой слой, все 47)
       └─ StatementImporter (только выручка) →  clients · projects · payments · acts
            └─ Domain Services               →  статусы актов, итоги, фильтры
                 └─ REST API (тонкие контроллеры + Resources)
                      └─ Vue 3 SPA (дашборд)
```

Backend (`backend/`) — Laravel 13, только JSON REST API. Frontend (`frontend/`) —
самостоятельное Vue 3 приложение, общается с API по HTTP. Связь между ними — только
сетевая, по контракту [`docs/API.md`](docs/API.md).

## Сущности и связи

| Сущность | Назначение | Ключевые поля |
|---|---|---|
| **client** | Юрлицо-плательщик (реквизиты из выписки) | name, type (ООО/АО/АНО/ИП), inn, ogrn, bank_* |
| **bank_operation** | Сырой слой: каждая операция выписки | op_date, direction, amount, purpose, category, is_revenue |
| **project** | Тело работ по клиенту (один на клиента) | client_id, name, status |
| **payment** | Доменная оплата (выручка по проекту) | project_id, client_id, payment_date, amount, work_direction, service_stage, invoice/contract_number |
| **act** | Закрывающий акт по оплате (1:1) | payment_id, is_sent, is_signed, status, manager_comment |

Связи: client 1—N project · client 1—N payment · project 1—N payment ·
payment 1—1 act · bank_operation 0/1—1 payment. Направление работ — атрибут оплаты
(один проект клиента может охватывать разные услуги).

## Где бизнес-логика

Вся доменная логика — в `backend/app/Domain` (не в контроллерах):

- **StatementParser / OperationClassifier / PurposeAnalyzer** — парсинг PDF, отделение
  выручки от шума, извлечение направления/счёта/договора/этапа.
- **StatementImporter** — доменная сборка (дедуп клиентов, проекты, оплаты, засев актов).
- **ActStatusService** — единый расчёт статуса акта.
- **DashboardSummaryService** — итоги на backend.
- **ActUpdater** — изменение акта с инвариантами.
- **PaymentFilter** (`app/Domain/Queries`) — фильтры на стороне БД.

Контроллеры (`app/Http/Controllers/Api`) тонкие: запрос → доменный сервис → API Resource.

## Логика статусов актов

Статус акта считается автоматически (`ActStatusService`):

- отправлен и подписан → **закрыт**;
- оплата старше N дней и акт не закрыт → **требует внимания**;
- отправлен, но не подписан → **ожидает подписи**;
- иначе → **не отправлен**.

«Сейчас» и порог давности — в `config/dashboard.php`. Поскольку данные выписки
датированы будущим (июль–август 2026), «сейчас» привязано к дате её формирования
(2026-08-14), порог — 21 день. В дашборде представлены все четыре статуса.

## API

Шесть эндпоинтов (полный контракт — [`docs/API.md`](docs/API.md)):

```
GET   /api/dashboard/summary     итоги (с учётом фильтров)
GET   /api/clients               юрлица с агрегатами
GET   /api/projects              проекты с агрегатами
GET   /api/payments              оплаты (пагинация, фильтры)
GET   /api/bank-operations       сырой слой выписки (прозрачность классификации)
PATCH /api/acts/{act}            отметить отправлен/подписан/комментарий
```

Фильтры (период, юрлицо, проект, направление, статус акта, поиск) считаются в БД и
общие для списка оплат и сводки.

## Принятые допущения

- Актов в выписке нет — слой статусов засеян детерминированным разбросом по возрасту
  оплаты (чтобы были видны все статусы); сами правила статуса — реальные.
- «Сейчас» привязано к дате выписки (2026-08-14), т.к. данные в будущем.
- Проект выводится как тело работ по клиенту; направление работ — на уровне оплаты.
- Небизнесовые операции (налоги/зарплаты/аренда/комиссии/депозиты/субподряд) исключены
  из выручки, но сохранены в сыром слое с категорией.
- Авторизация и роли не реализованы (по ТЗ — допустимо опустить).
- Контактное лицо клиента в выписке отсутствует — поле не заполняется.

## Стек

Laravel 13 (PHP 8.4) · Vue 3 + Vite + Pinia + Tailwind · PostgreSQL 16 (тесты — SQLite) ·
Docker Compose · smalot/pdfparser. Тестов: 35 (парсер, классификатор, статусы актов,
импорт, итоги, API).

## Структура

```
backend/    Laravel API (app/Domain — доменная логика, app/Http — API)
frontend/   Vue 3 SPA (src/views: Дашборд/Проекты/Клиенты/Выписка; src/components, src/stores)
docker/     nginx-конфиг и web-образ (сборка SPA + nginx)
docs/       PLAN.md, API.md, WORKLOG.md, brief/ (ТЗ + выписка)
```

Список оплат адаптивный: таблица на десктопе и карточки на узких экранах. Колонка
«Проект» в таблице не дублируется (проект 1:1 равен клиенту — он на странице «Проекты»
и в фильтрах). Статус акта показан плашкой с признаками «отправлен»/«подписан»;
комментарий менеджера и переключение статусов редактируются прямо из списка.
