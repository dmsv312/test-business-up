# План разработки — мини-система учёта оплат, проектов и закрывающих актов

Рабочий документ для самого процесса разработки. Дублирует ключевые архитектурные
решения, чтобы сборка шла связно и коммиты ложились логичными этапами. Часть этого
текста переедет в `README.md` (требование ТЗ: описание архитектуры, сущностей,
связей, где логика, какие допущения).

Исходное ТЗ: [`docs/brief/TASK.md`](brief/TASK.md).
Мок-данные: [`docs/brief/bank_statement_project_data_clean.pdf`](brief/bank_statement_project_data_clean.pdf).

## 1. Что оценивают (приоритеты)

ТЗ прямым текстом: интересует **не вёрстка, а бизнес-логика** — «как из хаотичных
оплат и документов сделать понятную систему контроля проектов, этапов и актов».
Критерии: понимание бизнес-задачи, качество модели данных, логика связей, чистота
кода, корректность расчётов/фильтров, **умение отделять данные / бизнес-логику /
отображение**, потенциал к развитию.

Вывод для архитектуры: упор на явные слои, доменные сервисы для расчётов, честный
парсер выписки, аккуратную модель. Визуал — простой, но опрятный.

## 2. Ключевой инсайт по данным (основа всего решения)

Источник — банковская выписка ИП ГРОМОВ А.В. (ИНН 782934761208) за период
15.07–14.08.2026, дата формирования **14.08.2026**. Всего **47 операций**
(21 списание + 26 поступлений).

**Не каждая операция — оплата клиента за проект.** Реальная выручка по проектам —
только **24 входящих платежа**. Остальное — шум, который система обязана отфильтровать:

- **списания (21):** налоги (НДФЛ, ЕНС), зарплаты, переводы ИП на личный счёт,
  аренда офиса/склада, банковские комиссии, депозиты/вклады, **оплаты субподрядчикам**
  (ИП Карпов, ИП Никольский, ООО «Команда на связи», ИП Руденко — это наши расходы);
- **поступления-нешум (2):** возврат депозита 810 000 ₽ и проценты банка 4 490 ₽.

Классификация проверена арифметикой и сходится в копейку с итогами выписки:

```
24 клиентских платежа         = 1 405 820,00 ₽
возврат депозита              =   810 000,00 ₽
проценты банка                =     4 490,00 ₽
-------------------------------------------------
итого кредит (из выписки)     = 2 220 310,00 ₽   ✓ совпадает
```

Это и есть демонстрация «из хаоса — система»: сырой слой хранит все 47 операций
с категорией, доменный слой работает только с 24 выручкой, в дашборде показываем
плашку «отфильтровано N небизнесовых операций».

Состав выручки: **19 клиентов/юрлиц**, у части по 2 оплаты (Сигма-Маркет,
Облако-Имидж, Светлый Квартал, Орбита-Промо, Ледник-Старт). По каждому плательщику
в выписке есть полный набор реквизитов: название, ИНН, ОГРН/ОГРНИП, расчётный счёт,
банк (БИК + наименование).

## 3. Стек и решения (согласовано с владельцем)

- **Backend:** Laravel 12 (PHP 8.4), только REST API (`routes/api.php`).
- **Frontend:** отдельный Vue 3 SPA (Vite + Pinia + Vue Router + Axios + Tailwind).
- **БД:** PostgreSQL 16 (в docker-compose); SQLite — для тестов.
- **Инфраструктура:** монорепо + `docker-compose.yml` (вся инфраструктура, включая БД).
- **Парсинг:** реальный парсер PDF выписки (`smalot/pdfparser`) → классификация → сид БД.
- **Сущность «Проект»** = тело работ по клиенту (одна запись на клиента); все оплаты
  клиента группируются в один проект, направление работ — атрибут оплаты.
- **Авторизация:** отсутствует (ТЗ разрешает опустить).
- **Деплой:** на свой сервер (Laravel Forge) — настраиваем в самом конце.

## 4. Архитектура слоёв

```
PDF-выписка
   │  StatementParser (smalot/pdfparser): строки → структурированные операции
   ▼
OperationClassifier: категория (tax/salary/rent/fee/deposit/owner_draw/
   │                            subcontractor/CLIENT_REVENUE) + is_revenue
   ▼
bank_operations (сырой слой, все 47 операций)
   │  доменная сборка (только is_revenue): client → project → payment → act
   ▼
clients · projects · payments · acts (доменный слой)
   │  Domain Services: ActStatusService, DashboardSummaryService, фильтры
   ▼
REST API (контроллеры тонкие, API Resources)
   ▼
Vue SPA (дашборд: итоги, фильтры, таблицы, переключение статусов актов)
```

Бизнес-логика — в `app/Domain/*` (сервисы/DTO/enum), не в контроллерах и не размазана
по моделям. Контроллеры только принимают запрос → зовут сервис → отдают Resource.

## 5. Модель данных

### clients (юрлицо / плательщик)
`id, name, type(enum: ooo/ano/ip), inn, ogrn, bank_account, bank_name, bank_bik,
timestamps`. Контактное лицо в выписке отсутствует — поле опускаем (или nullable).

### projects
`id, client_id(FK), name, status(enum: active/closed), timestamps`.
Проект = тело работ по клиенту (одна запись на клиента): все его оплаты группируются
в один проект. Направление работ — атрибут оплаты (проект может охватывать разные
услуги). `status` — производный (закрыт, если все акты проекта закрыты).

### bank_operations (сырой слой импорта, все 47)
`id, op_date, direction(enum: debit/credit), amount, doc_number,
counterparty_name, counterparty_inn, counterparty_account, purpose(text),
category(enum), is_revenue(bool), payment_id(nullable FK), timestamps`.

### payments (доменная выручка, только 24)
`id, project_id(FK), client_id(FK), bank_operation_id(FK), payment_date, amount,
payment_purpose(text), work_direction(nullable: направление работ),
service_stage(nullable: аванс/этап 1/финальный/…), invoice_number(nullable),
contract_number(nullable), timestamps`.

### acts (закрывающий документ по оплате)
`id, payment_id(FK), is_sent(bool), sent_at(nullable), is_signed(bool),
signed_at(nullable), status(enum), manager_comment(nullable text), timestamps`.

Связи: client 1—N project · client 1—N payment · project 1—N payment ·
payment 1—1 act · bank_operation 0/1—1 payment.

## 6. Доменная логика

### ActStatusService — авторасчёт статуса акта
- не отправлен и не подписан → **«не отправлен»** (`not_sent`)
- отправлен, не подписан → **«ожидает подписи»** (`awaiting_signature`)
- отправлен и подписан → **«закрыт»** (`closed`)
- оплата старше `N` дней и акт не закрыт → **«требует внимания»** (`needs_attention`)
  — перебивает первые два.

`«Сейчас»` = настраиваемая `REFERENCE_DATE` (config/env), по умолчанию **2026-08-14**
(дата выписки), т.к. данные датированы будущим. Порог `N` = `ACT_ATTENTION_DAYS`
(по умолчанию 30). Статус считается на лету при отдаче, плюс кэшируется в `acts.status`
после правок (для фильтрации в БД).

### DashboardSummaryService — итоги на backend
Общая выручка, число проектов, число оплат, сумма по закрытым актам, сумма по
незакрытым, число оплат без отправленного акта, число «отправлен, но не подписан».

### Фильтры (на стороне БД, в Query-объекте/scopes)
по проекту, по юрлицу, по периоду (from/to), по статусу акта, по направлению/типу
услуги, поиск по назначению платежа или названию клиента.

## 7. Парсер и классификация

`StatementParser` читает реальный PDF (`smalot/pdfparser`), режет на операции по
датам/блокам, извлекает: дату, сумму, № документа, обе стороны (плательщик/получатель
с ИНН/ОГРН/счётом/банком), назначение. Направление (debit/credit) определяем по тому,
наш ли счёт (40802810937184056213) стоит плательщиком.

`OperationClassifier` по ключам назначения проставляет категорию:

| категория       | ключи назначения                                              |
|-----------------|---------------------------------------------------------------|
| tax             | НДФЛ, ЕНС, единый налог, спецрежим                             |
| salary          | заработная плата, реестр … договор 4527…                      |
| owner_draw      | перевод средств предпринимателя на личный счёт                |
| rent            | аренда офиса, аренда склада                                    |
| bank_fee        | комиссия                                                       |
| deposit         | депозит, вклад, возврат депозита, проценты по депозит…        |
| subcontractor   | исходящий платёж ИП/ООО за услуги (разработка/прототипы/…)    |
| **client_revenue** | входящий платёж за услуги (разработка, SEO, реклама, …)    |

`is_revenue = (direction == credit && category == client_revenue)`.

### Вывод направления проекта (по ключам назначения)

| направление (enum)   | ключи                                                       |
|----------------------|-------------------------------------------------------------|
| development          | разработ, доработк сайт, лендинг, личный кабинет             |
| seo                  | SEO, продвижени                                             |
| context_ads          | Директ, контекст, рекламн кампани, ведение рекламы          |
| smm                  | SMM                                                         |
| serm                 | SERM                                                       |
| design               | дизайн, прототип                                            |
| content              | текст, копирайт, контент, наполнени, публикаци материал     |
| placement            | объявлени, размещени                                       |
| marketing            | маркетинг                                                  |
| support              | сопровождени                                               |
| presentation         | презентаци                                                 |
| other                | прочее (лицензии и т.п.)                                   |

Название проекта = `«{короткое имя клиента} — {человекочитаемое направление}»`.
Этап (`service_stage`) — из назначения: «этап 1», «аванс», «финальный платёж».

## 8. Сид актов (данных об актах в выписке нет)

Актов в источнике нет — слой моделируем детерминированным разбросом по возрасту
оплаты, чтобы в дашборде были видны все 4 статуса и работала автологика:
старые оплаты — преимущественно «закрыт»/«требует внимания», свежие — «не отправлен»/
«ожидает подписи». Допущение явно описываем в README. Разброс детерминированный
(seed по id/дате), без случайности — чтобы сборка воспроизводилась.

## 9. REST API

- `GET  /api/dashboard/summary` — итоги (с учётом активных фильтров).
- `GET  /api/clients` — юрлица с агрегатами (сумма, число оплат, закрытые/незакрытые акты, статус).
- `GET  /api/projects` — проекты с агрегатами + фильтры.
- `GET  /api/payments` — оплаты с фильтрами (проект, юрлицо, период, статус акта,
  направление, поиск), каждый с вложенным актом и его статусом.
- `PATCH /api/acts/{id}` — отметить отправлен/подписан, задать комментарий
  (изменения сохраняются в БД — требование ТЗ).
- `GET  /api/bank-operations` — сырой слой (для плашки «отфильтровано N операций»
  и демонстрации классификации).

Ответы — через API Resources. Валидация — Form Requests.

## 10. Frontend (Vue 3 SPA)

- **Страницы:** Дашборд (карточки-итоги + панель фильтров + таблица оплат),
  Проекты, Клиенты. Возможна отдельная вкладка «Выписка» (сырой слой).
- **Компоненты:** `SummaryCards`, `FiltersBar`, `PaymentsTable` (бейджи статусов +
  кнопки «отправлен»/«подписан» → `PATCH`), `ProjectsTable`, `ClientsTable`.
- **Состояние:** Pinia (стор фильтров + данных), Axios-клиент, debounce на поиск.
- Tailwind, адаптивная вёрстка (плюс по ТЗ), визуал намеренно простой.

## 11. Docker / запуск

`docker-compose.yml` поднимает: `app` (PHP-FPM Laravel), `nginx` (отдаёт API),
`db` (PostgreSQL 16), `frontend` (Vite dev / собранная статика через nginx).
Цель: `git clone` → `docker compose up` → миграции+сиды автоматически → дашборд
доступен. README с точными командами.

## 12. Тесты

- Feature-тесты API: фильтры, агрегаты, `PATCH` акта меняет статус и сохраняется.
- Unit-тесты доменных сервисов: `ActStatusService` (все 4 ветки + граница давности),
  `OperationClassifier` (выручка vs шум), `DashboardSummaryService` (суммы сходятся).
- Тест парсера: на реальном PDF получаем ровно 24 выручки и контрольную сумму
  1 405 820,00 ₽ (защита от регрессий классификатора).
- Прогон на SQLite.

## 13. Этапы → коммиты (примерный порядок)

1. **chore:** скелет монорепо, docker-compose, .gitignore, README, план.
2. **feat(backend):** Laravel skeleton, миграции, модели, enum, связи.
3. **feat(parser):** StatementParser + OperationClassifier + тесты на контрольную сумму.
4. **feat(seed):** сидеры из распарсенной выписки (clients/projects/payments/acts).
5. **feat(domain):** ActStatusService, DashboardSummaryService, фильтры + unit-тесты.
6. **feat(api):** контроллеры, ресурсы, Form Requests, маршруты + feature-тесты.
7. **feat(frontend):** Vue SPA — дашборд, фильтры, таблицы, переключение статусов.
8. **docs:** финал README (архитектура, сущности, связи, допущения), скриншоты.
9. **chore(deploy):** деплой на Forge, прод-конфиг.

## 14. Допущения (вести список для README)

- «Сейчас» для статусов актов привязано к дате выписки (2026-08-14), т.к. данные в будущем.
- Актов в источнике нет — статусы засеяны по модели «разброс по возрасту оплаты».
- Проект выводится из назначения платежа (договор/направление) — в источнике явного
  поля проекта нет.
- Небизнесовые операции (налоги/зарплаты/аренда/комиссии/депозиты/субподряд) исключены
  из выручки, но сохранены в сыром слое с категорией.
- Контактное лицо клиента в выписке отсутствует — поле не заполняется.
