# Журнал работ

Хронология сборки по этапам. Каждый этап — отдельный коммит (см. историю git).
План этапов — в [`PLAN.md`](PLAN.md) §13.

## Этап 1 — каркас репозитория и план ✅
- Инициализирован отдельный git-репозиторий, изолирован от родительского hh-bot.
- Перенесены ТЗ и мок-данные в `docs/brief/`.
- Написаны `docs/PLAN.md` (подробный план + архитектура), `README.md`, `.gitignore`.
- Зафиксирован инсайт по данным (24 выручки из 47 операций, контрольная сумма сходится).

## Этап 2 — каркас backend и доменная схема ✅
- Laravel 13.17 в `backend/`, подключён API-роутинг (без Sanctum/авторизации).
- Слой `app/Domain/Enums`: ClientType, ProjectDirection, ProjectStatus,
  OperationDirection, OperationCategory, ActStatus (все с `label()` на RU).
- Миграции доменной схемы: clients, bank_operations, projects, payments, acts.
  Цикл `bank_operation ↔ payment` разорван (FK только в `payments.bank_operation_id`).
- Модели Eloquent со связями и кастами enumّов.
- `config/dashboard.php`: reference_date (2026-08-14), act_attention_days (30), own_account.
- `.env.example` переведён на PostgreSQL (цель docker); локально/тесты — SQLite.
- Проверено фактом: миграции прогоняются на sqlite, роут `api/ping` поднимается,
  enum'ы и конфиг грузятся.

## Этап 3 — парсер выписки и классификатор ✅
- `smalot/pdfparser` v2.12.5; PDF выписки положен в `backend/database/data/bank_statement.pdf`.
- `StatementParser` (PDF → `ParsedOperation[]`): нарезка по строкам-датам, разбор сторон
  по 20-значным счетам, направление по нашему счёту, разбор банка и назначения из
  таб-разделённого «хвоста».
- `OperationClassifier`: категория (выручка/налог/зарплата/вывод ИП/аренда/комиссия/
  депозит/субподряд). `PurposeAnalyzer`: направление работ, № счёта, № договора, этап.
- Тесты (`tests/Feature/StatementParserTest`, 7 кейсов): контрольная сумма
  **24 операции = 1 405 820,00 ₽**, плюс сверка оборотов с «Итого оборотов» выписки
  (дебет 2 437 811,53 / кредит 2 220 310,00) — доказывает полноту и верность направлений.
- **Состязательная проверка** (workflow `verify-classifier`, 4 аудитора-скептика +
  судьи по каждому замечанию): из 47 операций × 4 аспекта подтверждено одно фактическое
  замечание — потеря второго/третьего № счёта в «по счетам № 738, 791 и 792» (оп. 43).
  Исправлено (`invoiceNumber` собирает все номера) + регрессионный тест. Границы
  выручка/шум, категории и направления работ замечаний не вызвали.

## Этап 4 — доменная сборка (сидеры) ✅
- `ActStatusService` — единый расчёт статуса акта (закрыт / ожидает / не отправлен /
  требует внимания) от is_sent/is_signed/возраста оплаты. Порог «требует внимания»
  снижен до 21 дня (иначе при дате отсчёта 14.08 он не срабатывал). Вынесен сюда из
  этапа 5, т.к. без него не засеять акты.
- `StatementImporter` — из `ParsedOperation[]`: сырой слой (47), клиенты (дедуп по ИНН),
  проекты (один на клиента), оплаты (24), засев актов. Производный статус проекта
  (закрыт, если все акты подписаны). Добавлена форма АО в `ClientType`.
- Точки входа: команда `statement:import {path?}` + `StatementSeeder` (в `migrate --seed`).
- Тесты (22 всего, 119 проверок): юнит `ActStatusService` (все ветки + граница давности),
  feature импорта (47 операций, 19 клиентов, 24 оплаты/акта, сумма 1 405 820,00 ₽,
  все 4 статуса актов, дедуп клиентов, статус проекта, имя/группировка проекта).

Решение по группировке (после обсуждения с владельцем): **проект = клиент** (одна запись
на клиента), направление работ перенесено на оплату (`payments.work_direction`). Итог:
19 проектов, из них 5 мультиоплатных (Облако, Сигма, Светлый, Орбита, Ледник) — связь
проект→оплаты теперь наглядна. Схема: `projects` без direction/contract; они на оплате.

## Этап 5 — итоги и фильтры ✅
- `PaymentFilter` (`app/Domain/Queries`) — единый слой фильтров на стороне БД
  (период, клиент, проект, направление, статус акта, поиск), общий для списка
  оплат и сводки.
- `DashboardSummaryService` — итоги на backend (выручка, счётчики, суммы по
  закрытым/открытым актам, «требует внимания», число отфильтрованного шума);
  деньги в копейках. Тесты: итоги, сходимость закрыто+открыто, фильтры.

## Этап 6 — REST API ✅
- 6 эндпоинтов под `docs/API.md`: `GET dashboard/summary · clients · projects ·
  payments (пагинация) · bank-operations`, `PATCH acts/{act}`.
- Тонкие контроллеры (`app/Http/Controllers/Api`), API Resources (общий трейт
  агрегатов), Form Requests (`DashboardFilterRequest` с валидацией enum-фильтров,
  `UpdateActRequest`). `ActUpdater` — обновление акта с инвариантами + пересчёт
  статуса; изменения сохраняются.
- Feature-тесты (через HTTP-стек): каждый эндпоинт, фильтры, пагинация, `PATCH`
  закрывает акт и персистит, валидация даёт 422. Всего 35 тестов / 186 проверок.

## Этап 7 — Vue 3 SPA ✅
- Отдельное приложение `frontend/` (Vite 8, Vue 3.5, Router 5, Pinia 3, Axios,
  Tailwind 4). Связь с backend только по HTTP; dev — через Vite-прокси на `/api`.
- Pinia-стор `dashboard`: фильтры + данные; `loadDashboard`/`updateAct` и пр.
- Страницы: Дашборд (карточки-итоги, панель фильтров, таблица оплат с переключением
  статусов актов через `PATCH`, пагинация), Проекты, Клиенты, Выписка (сырой слой
  с подсветкой выручки и баннером «24 из 47 — выручка»).
- Компоненты: `AppNav`, `SummaryCards`, `FiltersBar`, `PaymentsTable`, `ActStatusBadge`.
- Проверено: `npm run build` собирается; API поднят локально и отвечает корректным
  JSON (формы совпадают с тем, что потребляют компоненты). Визуальный e2e — на этапе
  docker (полный стек одной командой).
